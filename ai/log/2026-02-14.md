- Performance analysis: SELECT * FROM t is ~6x slower than native PG.
  Identified 7 bottlenecks in the IPC hot path (see dec-0009):
  1. SIGUSR1 ping-pong per page (~5-20 us per signal, 2000+ signals per scan)
  2. Backend processes one heap request per exec_df_scan iteration
  3. wait_latch(1ms) adds baseline latency when result ring is empty
  4. relation_open/relation_close called per page instead of per scan
  5. Worker loop processes one message per wake, signals after each
  6. Full Arrow round-trip per row (heap -> ScalarValue -> Arrow -> wire tuple -> MinimalTuple)
  7. Result ring 64KB too small for burst writes

- Primary suspicion confirmed: signal wakeup latency is a major contributor,
  but the single-message-per-wake pattern in both worker and backend amplifies it.
  The combination of per-page SIGUSR1 + 1ms wait_latch + one-at-a-time processing
  creates a multiplicative slowdown.

- Proposed fix order: batch processing in worker/backend loops first (biggest bang),
  then remove 1ms wait_latch, then cache relation, then increase ring size.
