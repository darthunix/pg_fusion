- Main code hotspots confirmed:
  - backend processes max one heap request per loop iteration:
    - `postgres/src/backend.rs` (`process_pending_heap_request` call site in `exec_df_scan`)
  - frequent `wait_latch(Some(1ms))` in hot path:
    - `postgres/src/backend.rs`
  - per-page `relation_open/relation_close`:
    - `postgres/src/backend.rs`
  - small result ring (`64 KB`):
    - `postgres/src/worker.rs` (`RESULT_RING_CAP`)
  - execution path sends `SIGUSR1` after each written batch:
    - `executor/src/server.rs`

- Conclusion:
  - Signal ping-pong and wakeup cadence are real and significant.
  - Additional overhead exists beyond signals (result ring path, tuple encode/decode, Arrow/DataFusion path).
  - There is also a correctness mismatch that should be investigated in parallel with performance work.
