# pg_statistics

`pg_statistics` is a PostgreSQL-specific statistics bridge for pg_fusion
planning. It does not depend on DataFusion or `join_order`.

The crate exposes three kinds of inputs:

- PostgreSQL planner estimates for one scan SQL string, without executing it
- catalog statistics for relation rows, column NDV/null/width, and unique keys
- deterministic helpers that combine those values into equi-join selectivity

`plan_builder` can use these values later to build compact `join_order`
problems where leaf `RelStats` already reflect local pushed filters and join
edges have PostgreSQL-aware selectivity estimates.

## Scan Estimates

`estimate_scan_sql(sql, options)` prepares the SQL through SPI, gets the cached
PostgreSQL plan, and reads typed fields from `PlannedStmt.planTree`.

It returns `PgScanEstimate`:

- `rows`: PostgreSQL's estimated output rows after local scan predicates
- `width`: estimated row width in bytes
- `bytes`: `rows * width`, useful as a compact payload-size signal
- `startup_cost`: PostgreSQL planner startup cost
- `total_cost`: PostgreSQL planner total cost

The current implementation reads:

- `Plan.plan_rows`
- `Plan.plan_width`
- `Plan.startup_cost`
- `Plan.total_cost`

It does not parse `EXPLAIN` text and does not execute the query.

`EstimateOptions` controls planning flags passed to PostgreSQL:

- `parallel_ok`: sets `CURSOR_OPT_PARALLEL_OK` so PostgreSQL can consider
  parallel-safe plans while estimating
- `fast_start`: sets `CURSOR_OPT_FAST_PLAN` when the caller wants a
  fast-start bias

This function is intended for scan SQL generated by pg_fusion. It requires a
single SELECT statement that PostgreSQL can prepare as a cursor plan.

## Catalog Statistics

`load_relation_stats(relation_oid)` reads `pg_class` through syscache and
returns `PgRelationStats`:

- `rows`: `pg_class.reltuples`, when PostgreSQL has a non-negative estimate
- `pages`: `pg_class.relpages`, when PostgreSQL has a non-negative estimate

`load_column_stats(relation_oid, attnums)` validates each attribute and reads
`pg_statistic`. It prefers inherited statistics when present, which is important
for partitioned parents, and falls back to non-inherited stats otherwise. Each
`PgColumnStats` contains:

- `attnum`: relation attribute number
- `inherited`: whether the chosen `pg_statistic` row was inherited
- `null_frac`: `pg_statistic.stanullfrac`
- `avg_width`: `pg_statistic.stawidth`
- `stadistinct`: raw `pg_statistic.stadistinct`
- `ndv`: normalized number of distinct values

`stadistinct` normalization follows PostgreSQL's catalog convention:

- positive values are exact estimated NDV
- negative values mean `-stadistinct * reltuples`
- zero or missing relation rows produce unknown NDV

`PgColumnStats::effective_ndv(filtered_rows)` caps base NDV by the filtered leaf
row estimate. This is the value join planning should normally use for a scan
leaf after local predicates.

`load_unique_keys(relation_oid)` scans relation index metadata and returns
`PgUniqueKey` entries for valid, live unique indexes whose key attributes are
plain relation columns. Expression indexes are skipped for v1 because there is
no compact column attnum set to feed into the join-order problem.
Partial unique indexes are skipped too: their uniqueness holds only under the
index predicate, so they are not safe relation-wide uniqueness signals unless a
future caller can prove the scan predicate implies the index predicate.

## Join Selectivity

`estimate_equi_join_selectivity` is a deterministic v1 estimator for one
equality join predicate. It uses caller-provided filtered leaf row counts and
column NDV/null fractions from PostgreSQL catalog stats. Unknown NDV falls back
to a conservative fixed selectivity.

For known NDV on both sides, the estimator uses the common formula:

```text
nonnull(left) * nonnull(right) / max(ndv(left), ndv(right))
```

When only one NDV is known, it uses the known NDV and marks quality as
`Partial`. When neither side is known, it uses the fixed fallback selectivity
and marks quality as `Fallback`.

Unique-key flags clamp the output cardinality:

- right side unique: output cannot exceed non-null left rows
- left side unique: output cannot exceed non-null right rows
- both sides unique: output cannot exceed the smaller non-null side

The result is `EquiJoinEstimate`:

- `selectivity`: estimated output rows divided by Cartesian input rows
- `rows`: estimated join output rows
- `quality`: `Full`, `Partial`, or `Fallback`

## Testing

The crate references PostgreSQL backend symbols, so standalone `cargo test -p
pg_statistics` is not the primary test path. PostgreSQL-facing coverage lives in
`pg/test`:

```sh
cargo pgrx test pg17 -p pg_test
```

Use `cargo check -p pg_statistics` for fast type checking outside PostgreSQL.
